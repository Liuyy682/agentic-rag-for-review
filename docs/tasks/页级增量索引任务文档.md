# 页级增量索引任务文档

## 背景

当前项目的文档入库流程是：

```text
PDF / Markdown
  -> Markdown 文件
  -> Markdown 清洗
  -> parent chunk
  -> child chunk
  -> Qdrant 向量库
  -> parent store
```

现有实现对已存在的 Markdown 文件会直接跳过，因此同名文档覆盖更新后无法自动刷新索引。如果采用删除整个文档再重建的方式，可以避免全库重建，但对单个文档来说仍然是文档级全量更新。

本任务目标是在同名文档覆盖更新时，新增 **页级增量索引**：只识别发生变化的页面，并只重建与这些页面相关的 parent / child chunks，避免对未变化页面重复转换、分块、embedding 和写库。

## 已确认约束

1. 最小更新单元使用 **页**，不使用章节。

2. 文档更新方式为 **覆盖同名文件**。

3. PDF 转 Markdown 使用 `pymupdf4llm`，转换结果认为稳定。

4. 图片、表格、OCR、图片描述等内容只要进入 Markdown 正文，就参与页面 hash；如果它们变化，视为对应页面变化。

5. 允许重建变更页相邻区域，尤其是跨页合并或 chunk overlap 影响到的 chunk。

6. 如果 chunker 配置、清洗规则、embedding 模型或索引 schema 变化，按配置版本判断为局部增量不可信，应提示或触发更大范围重建。

7. 更新行为自动触发：上传或导入同名文档时自动检测并增量更新。

8. manifest 按项目数据存储，建议放在 `parent_store/index_manifest.json`，也可以独立放在数据目录下。

9. 不保留历史版本。

10. 本任务只生成实现任务文档，不立即修改 UI。

## 总体目标

新增页级增量索引能力，使系统在同名文档更新后能够自动完成局部刷新。

核心目标：

- 为每个文档维护页面 hash manifest。
- 上传同名文档时不再直接跳过，而是检测页面变化。
- 只重建发生变化的页面及必要相邻页面。
- 删除 Qdrant 中与受影响页面重叠的旧 child chunks。
- 删除 parent store 中与受影响页面重叠的旧 parent chunks。
- 对受影响页面重新清洗、分块、embedding、写库。
- 更新 manifest。
- chunker / cleaner / embedding 配置变化时拒绝错误的局部增量。

## 非目标

本任务第一阶段不处理以下内容：

- 不做 chunk_index 级别的一一对齐更新。
- 不做复杂文本 diff 或段落移动检测。
- 不保留文档历史版本。
- 不新增 UI 页面或按钮。
- 不改变检索、rerank、Agent 编排逻辑。
- 不改变 PDF 转 Markdown 工具选型。
- 不在配置变化时强行做不可靠的局部更新。

## 设计原则

### 不按 chunk 顺序对齐

chunk 每次切分出来可能不同，尤其是在页面开头插入内容、清洗规则变化、chunk overlap 变化时，后续 chunk 的编号和边界都可能偏移。

因此不能使用如下逻辑判断更新：

```text
这次第 8 个 chunk == 上次第 8 个 chunk
```

正确策略是：

```text
按稳定页面识别变化
  -> 找到受影响页面范围
  -> 删除旧索引中 page_numbers 与该范围重叠的 chunks
  -> 对该范围重新分块并写入新 chunks
```

### 页面是稳定源单元

`pymupdf4llm` 转换后的 Markdown 已包含页面边界。页面应作为 manifest 中的稳定源单元。

页面 hash 应基于进入 RAG 前的有效页面内容，建议使用清洗后的 page text，因为最终入库内容来自清洗结果。

如果图片描述、OCR、表格文本在清洗后页面内容中发生变化，也应导致该页 hash 变化。

### 允许局部扩大影响范围

parent chunk 可能跨页合并，child chunk 也可能因 overlap 影响相邻内容。为了避免保留过期 chunk，增量更新不能只删除“变化页本身”。

建议影响范围：

```text
affected_pages = changed_pages + previous_page + next_page
```

如果 parent chunk metadata 中的 `page_numbers` 与 `affected_pages` 有交集，则删除该 parent 及其 child chunks。

## 推荐处理流程

同名文档导入流程：

```text
收到 document_path
  -> 转换或覆盖 markdown_docs/{doc_stem}.md
  -> 基于 pymupdf4llm page marker 切页
  -> 执行现有 Markdown 清洗
  -> 计算每页 cleaned_text hash
  -> 读取 index_manifest
  -> 判断索引配置是否兼容
      -> 不兼容：提示需要全量重建或走文档级重建
      -> 兼容：继续页级 diff
  -> 找 changed_pages
      -> 无变化：跳过
      -> 有变化：计算 affected_pages
  -> 删除旧索引中与 affected_pages 重叠的 parent / child chunks
  -> 只对 affected_pages 对应页面重新生成 parent / child chunks
  -> 写入 Qdrant 和 parent store
  -> 更新 manifest
```

首次导入流程：

```text
文档未出现在 manifest
  -> 按现有方式完整切分该文档
  -> 写入 Qdrant 和 parent store
  -> 记录文档级 hash、页面 hash、parent ids、child ids
```

## Manifest 设计

建议新增 manifest 文件：

```text
parent_store/index_manifest.json
```

建议结构：

```json
{
  "schema_version": 1,
  "index_config": {
    "dense_model": "BAAI/bge-large-en-v1.5",
    "sparse_model": "Qdrant/bm25",
    "chunker_config_hash": "...",
    "cleaner_config_hash": "...",
    "embedding_config_hash": "..."
  },
  "documents": {
    "example.pdf": {
      "doc_id": "example",
      "source_file": "example.pdf",
      "markdown_path": "markdown_docs/example.md",
      "document_hash": "...",
      "pages": {
        "1": {
          "page_hash": "...",
          "parent_ids": ["example_parent_0"],
          "child_ids": ["example_parent_0_child_0", "example_parent_0_child_1"]
        }
      },
      "parent_pages": {
        "example_parent_0": [1, 2],
        "example_parent_1": [3]
      },
      "updated_at": "2026-05-08T00:00:00Z"
    }
  }
}
```

说明：

- `documents` 使用 `source_file` 作为 key，因为用户确认更新是覆盖同名文件。
- `doc_id` 可以继续使用 `Path(doc_path).stem`。
- `pages` 记录每页 hash 和由该页产生或关联的 chunk id。
- `parent_pages` 用于快速判断哪些 parent chunks 与变更页重叠。
- `document_hash` 用于快速判断整篇文档完全无变化，但不能替代 page hash。
- `index_config` 用于判断局部增量是否可信。

## 配置版本判断

局部增量依赖稳定的清洗、分块和 embedding 策略。建议 manifest 中记录配置 hash。

配置 hash 应至少包含：

```text
DENSE_MODEL
SPARSE_MODEL
SPARSE_VECTOR_NAME
MARKDOWN_CLEANING_ENABLED
HEADER_FOOTER_SCAN_LINES
MIN_REPEAT_PAGES
MIN_REPEAT_RATIO
HEADERS_TO_SPLIT_ON
MIN_PARENT_SIZE
MAX_PARENT_SIZE
CHILD_CHUNK_SIZE
CHILD_CHUNK_OVERLAP
```

当当前配置 hash 与 manifest 不一致时：

```text
禁止页级增量
  -> 返回明确提示：索引配置已变化，需要全量重建
```

如果产品上希望自动处理，也可以退化为文档级重建；但第一阶段建议明确提示，避免隐式产生混合版本索引。

## 删除策略

### Parent store 删除

需要新增按 `parent_id` 删除的方法。

删除条件：

```text
parent_pages[parent_id] 与 affected_pages 有交集
```

删除动作：

```text
删除 parent_store/{parent_id}.json
```

### Qdrant 删除

需要新增按 metadata filter 删除 child chunks 的方法。

推荐优先按 `parent_id` 删除：

```text
child.metadata.parent_id in stale_parent_ids
```

如果当前 Qdrant / LangChain 封装不方便做 in filter，可以逐个 parent_id 删除。

备选删除条件：

```text
source_file == 当前文档
and page_numbers 与 affected_pages 有交集
```

推荐 parent_id 删除的原因是 parent / child 关系更直接，且可以避免 list metadata 过滤兼容性问题。

## 局部重建策略

### 基本要求

受影响页面重建时，不能直接复用整篇文档的 parent index 编号，否则局部重建后 parent id 可能与未更新页面冲突。

建议第一阶段采用稳定且简单的页级 parent id：

```text
{doc_id}_page_{page_number}_parent_{local_parent_index}
{doc_id}_page_{page_number}_parent_{local_parent_index}_child_{local_child_index}
```

如果一个 parent chunk 跨多个页面，可使用起始页作为 id 主体，并在 metadata 中保留完整 `page_numbers`。

示例：

```text
example_page_12_parent_0
example_page_12_parent_0_child_0
```

### 与现有 ID 的兼容

当前代码使用：

```text
{doc_stem}_parent_{i}
{doc_stem}_parent_{i}_child_{child_index}
```

这是全篇顺序编号，不适合局部增量。

本任务需要调整 chunk id 生成逻辑，使其支持页级稳定 id。首次导入也应使用新 id 规则，避免同一文档混用两套 id。

如果已有旧索引存在，建议在首次启用页级增量时要求用户全量重建一次，使 manifest 和新 id 规则一致。

## Metadata 要求

写入 parent 和 child chunks 时，metadata 至少应包含：

```text
source
source_file
doc_id
parent_id
chunk_id
page_number
page_numbers
page_hashes
index_schema_version
chunker_config_hash
cleaner_config_hash
embedding_config_hash
```

其中：

- `page_number` 可使用 parent 起始页或主要页。
- `page_numbers` 必须保留完整覆盖页列表。
- `page_hashes` 记录 chunk 覆盖页面的 hash，便于排查。
- `chunk_index` 可继续保留用于展示或排序，但不能作为增量更新依据。

## 错误处理

建议处理以下场景：

### Manifest 不存在

视为首次索引，完整导入该文档并创建 manifest。

### Manifest 损坏

不要继续局部增量。应返回明确错误，提示 manifest 损坏，需要备份后全量重建。

### Qdrant 删除失败

不要继续写入新 chunks，避免新旧索引混杂。

### Parent store 删除失败

不要继续写入新 parent chunks，避免 parent_id 内容不一致。

### 写入新 chunks 失败

manifest 不应更新。允许下次重试。

### Manifest 写入失败

索引可能已经更新，但 manifest 未更新。应记录错误并提示用户重新执行同步或全量重建。

## 原子性建议

第一阶段不需要实现完整事务，但应尽量保证失败后可恢复：

```text
1. 先计算 diff，不修改状态。
2. 删除旧 Qdrant points。
3. 删除旧 parent store 文件。
4. 写入新 Qdrant points。
5. 写入新 parent store 文件。
6. 最后原子写 manifest。
```

manifest 写入建议：

```text
写 index_manifest.json.tmp
fsync / close
rename 为 index_manifest.json
```

## 测试计划

### 单元测试

新增测试覆盖：

1. 页面 hash 未变化时跳过更新。

2. 单页变化时只返回该页及相邻页作为 `affected_pages`。

3. `parent_pages` 与 `affected_pages` 有交集时能正确找出 stale parent ids。

4. 配置 hash 变化时禁止页级增量。

5. manifest 原子读写正常。

6. parent store `delete_many` 只删除指定 parent 文件。

7. Qdrant 删除接口使用正确 metadata filter。

### 集成测试

建议构造一个 3 页 Markdown fixture：

```text
page 1 unchanged
page 2 changed
page 3 unchanged
```

验证：

- 首次导入写入 manifest。
- 第二次导入同名文件，只有 page 2 变化。
- `affected_pages` 为 `{1, 2, 3}`，因为需要包含相邻页。
- 旧 parent ids 被删除。
- 新 parent / child chunks 被写入。
- manifest 中 page 2 hash 更新。
- 未变化页面 hash 保持不变。

另构造一个 5 页 fixture：

```text
page 3 changed
```

验证：

- `affected_pages` 为 `{2, 3, 4}`。
- page 1 和 page 5 对应的 parent chunks 不被删除。

## 验收标准

完成后应满足：

1. 同名文档覆盖上传时，系统自动检测页面变化。

2. 未变化文档不会重复 embedding。

3. 单页变化不会触发整篇文档所有页面重建。

4. 变更页及相邻页相关旧 chunks 会被删除并重建。

5. Qdrant 中不会残留同一受影响区域的新旧重复 chunks。

6. parent store 中不会残留已删除 parent id 的旧内容。

7. manifest 能准确记录页面 hash、parent ids、child ids 和配置版本。

8. chunker / cleaner / embedding 配置变化时不会执行不可靠的局部增量。

9. 现有检索、rerank、Agent 回答流程不受影响。

10. 测试能覆盖首次导入、无变化跳过、单页变化、配置变化阻断和删除失败场景。

## 建议实施顺序

1. 新增 manifest 管理模块和单元测试。

2. 为 parent store 增加 `delete_many`。

3. 为 vector db 增加按 `parent_id` 删除 points 的能力。

4. 调整 chunker，使其支持页级稳定 id 和页面子集重建。

5. 调整 `DocumentManager.add_documents()`，同名文档自动进入页级增量流程。

6. 补充集成测试，验证 Qdrant、parent store、manifest 三者一致性。

7. 手动用一个多页 PDF 覆盖更新验证实际链路。
